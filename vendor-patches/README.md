# Patches to the vendored snapshot (vendor/art, vendor/libcore)

`vendor/` holds art + libcore at **android-16.0.0_r4**, cloned by
`tools/vendor-sync.sh` (gitignored — not committed; ~260 MB upstream). These are
project-owned (NOT the read-only archive), so we may patch them. The patches
below adapt the AOSP sources to the host clang-21 / libc++ toolchain. They are
minimal and behavior-preserving. `tools/vendor-sync.sh` re-applies them after a
fresh clone (each is grep-guarded, so an upstream-fixed patch self-skips).

When the toolchain or snapshot changes, re-evaluate; upstream AOSP has since
fixed some of these.

## Status at android-16.0.0_r4 (bumped from android-u-beta-4-gpl, 2026-06-21)

- **OBSOLETE (fixed upstream / file removed)** — no longer applied:
  - 0001 FillVRegs ambiguity: A16 rewrote FillVRegs to a single overload with
    `if constexpr (sizeof...(args) > 0)` recursion. No ambiguity.
  - 0007 dexlayout Iterator::operator- const: the entire `dexlayout/` directory
    was removed in A16 (only dexdump/dexlist remain).
  - 0008 register_allocator RemoveIntervalAndPotentialOtherHalf return type:
    upstream refactored the function away.
  - 0010 InvokeStatic/Instance arg byte count: A16 now passes
    `vregs.empty() ? nullptr : vregs.data()` + `vregs.size()*sizeof(value_type)`
    — exactly this fix, upstream.
- **STILL APPLIED**: 0002 (guard updated for the new `[[maybe_unused]] nullptr_t`
  form), 0003, 0004, 0005, 0009, 0011-0013 (host-toolchain shims), 0014
  (InlinedVector size_ init).
- **Non-patch coherence shims** (NOT source edits; in vendor-sync / build):
  - `vendor/nativehelper-headers/`: A16 libnativehelper header-only headers
    (utils.h + ScopedUtfChars' `operator std::string_view()`) the archive's 2023
    libnativehelper lacks; vendor-sync fetches them, the build puts them first on
    art's include path.
  - aconfig flag headers (`com_android_art[_rw]_flags.h`) generated by
    bp2cmake.aconfig from `art/build/flags/*.aconfig` (no aconfig tool on host).
  - `//compat/include/android-base/stringify.h` (archive libbase lacks it).


## 0001 art/runtime/art_method-inl.h — FillVRegs overload ambiguity (clang>=17)

The terminal `FillVRegs` template took variadic value params identical (after
pack expansion) to the recursive overload; with `ShortyTraits<>::Type` being a
non-deduced context, clang>=17 cannot pick a more-specialized overload -> hard
"call to 'FillVRegs' is ambiguous". Fix: drop the value params from the terminal
overload (it ignored them anyway):

    template <char... ArgType>
    inline ALWAYS_INLINE void FillVRegs(uint32_t* vregs ATTRIBUTE_UNUSED)
        REQUIRES_SHARED(Locks::mutator_lock_) {}

## 0002 art/runtime/class_linker.cc — bare nullptr_t (clang>=?, strict)

`Finish(nullptr_t np ...)` uses the unqualified `nullptr_t`; newer clang requires
`std::nullptr_t`. Fix: qualify it.

    ObjPtr<mirror::Class> Finish(std::nullptr_t np ATTRIBUTE_UNUSED)

## 0003 art/runtime/thread_linux.cc — constexpr with MINSIGSTKSZ (glibc>=2.34)

glibc 2.34 made `MINSIGSTKSZ` a non-constant `sysconf()` call, so
`static constexpr int kHostAltSigStackSize = ... MINSIGSTKSZ ...` no longer
compiles. Fix: drop `constexpr` (runtime-initialized const):

    static const int kHostAltSigStackSize = 32 * KB < MINSIGSTKSZ ? ... ;

(The sibling `PTHREAD_STACK_MIN` -Wsign-conversion in thread.cc is patch 0004.)

## 0004 art/runtime/thread.cc — PTHREAD_STACK_MIN long->size_t (glibc>=2.34)

`PTHREAD_STACK_MIN` is a non-const `long` (sysconf) on glibc>=2.34; thread.cc has
`#pragma clang diagnostic error "-Wconversion"` at file scope, so assigning it to
`size_t stack_size` is a hard error NO command-line flag can override (a pragma
beats `-Wno-error`/`-Wno-sign-conversion`). Fix: cast explicitly.

    if (stack_size < static_cast<size_t>(PTHREAD_STACK_MIN)) {
      stack_size = static_cast<size_t>(PTHREAD_STACK_MIN);

## 0005 art/openjdkjvm/OpenjdkJvm.cc — JVM_IsNaN bare isnan (clang>=?, libc++)

`JVM_IsNaN` calls unqualified `isnan(d)`. `<cmath>` puts it in `std::` only, and
we cannot force `<math.h>` globally because glibc-2.40's `<math.h>` declares an
IEEE-754 `canonicalize()` that collides with OpenJDK's own `canonicalize` in
jni_util.c / canonicalize_md.c / UnixFileSystem_md.c. Fix: qualify the one call.

    return std::isnan(d);

## 0006 — REMOVED (was String::ClassSize +67 -> +66)

This patch forced `String::ClassSize` from the upstream `kVTableLength + 67` to
`+66` to match a String that linked at 771 vtable bytes. That 771 was itself a
SYMPTOM of the uninitialized-BitVector bug now fixed in **0009**: the buggy
linker mis-built String's vtable one slot short. With 0009 in place, String
links at the canonical 779 and the upstream `+67` is correct, so this patch is
gone — re-applying it would re-break `CheckSystemClass(String)`. Leave
`string-inl.h` at upstream `+67`.

## 0007 art/dexlayout/dex_ir.h — Iterator::operator- must be const (libstdc++15)

`dex_ir::Iterator::operator-(const Iterator&)` was non-const; libstdc++-15's
`std::stable_sort` (used in dexlayout.cc) computes `__middle - __first` on
`const` iterators, so it needs a const difference operator. Add `const`.

    difference_type operator-(const Iterator& rhs) const { ... }

## 0008 art/compiler/optimizing/register_allocator_linear_scan.cc — iterator type (libstdc++15)

`RemoveIntervalAndPotentialOtherHalf` was declared returning
`ArenaVector<LiveInterval*>::iterator` but operates on a `ScopedArenaVector`
(different allocator adapter). Old libstdc++ silently converted the two
`__normal_iterator` types; libstdc++-15 rejects it. Fix the return type to
`ScopedArenaVector<LiveInterval*>::iterator`.

## 0009 art/runtime/class_linker.cc — uninitialized BitVector -> LinkMethods hang

`LinkMethodsHelper::AssignVTableIndexes` builds a `BitVector initialized_methods`
over an **uninitialized** `alloca`/arena buffer using the preallocated-storage
`BitVector` ctor, which (unlike the allocating ctor) does NOT clear the storage.
The algorithm assumes the bits start at 0 (it only `SetBit()`s and tests).
Garbage bits make `IsBitSet()` spuriously true on a super slot's first
encounter, so `same_signature_vtable_lists[j]` is set to a freshly-loaded
`ArtMethod`'s `method_index` (0) — a self-loop (`list[0]==0`). The later vtable
walk then spins forever (the guarding `DCHECK_LT` is compiled out under
`-DNDEBUG`). Symptom: dex2oat / imageless dalvikvm hang at 100% CPU linking
`java.util.HashMap`. Latent upstream bug, masked when the page is already zero.
Fix: zero the buffer right after construction.

    std::fill_n(bit_vector_buffer_ptr, bit_vector_size, 0u);

## 0010 art/runtime/art_method-inl.h — invoke-stub arg size (libstdc++ std::array<T,0>)

`ArtMethod::InvokeStatic`/`InvokeInstance` passed `sizeof(vregs)` as the arg
byte count to the invoke stub. For a no-arg static call (e.g. `Daemons.start()`
from `Runtime::Start`/`StartDaemonThreads`), `vregs` is `std::array<uint32_t,0>`:
under libstdc++ (host) `sizeof()==1` and `data()==nullptr`, so the stub did
`rep movsb` of 1 byte from NULL — SIGSEGV in `art_quick_invoke_static_stub`.
libc++ (AOSP) returns a non-null `data()`, so upstream never hit it. Fix: pass
the true byte count, which is 0 for no-arg and identical (`N*4 == sizeof`)
otherwise. Two call sites.

    Invoke(self, vregs.data(), vregs.size() * sizeof(uint32_t), &result, shorty.data());

## 0014 libartbase/base/inlined_vector.h — uninitialized size_

`InlinedVector` (new in A16) collects values in a small stack array, spilling to
`std::vector` past `kMaxStackEntries`. Its `size_` member had no initializer and
no user-defined constructor, so default construction left `size_` as stack
garbage. `Class::GetDeclaredFields` uses it for ConcurrentHashMap's 37 fields
(> 8), so `GetArray()` returned a garbage length and `CreateFromArtField` saw
null `ArtField*` pointers — SIGSEGV at `CreateFromArtField+66` during
`ConcurrentHashMap.<clinit>` / `Unsafe.objectFieldOffset`. Adding `fprintf` in
the loop "fixed" it by zeroing stack (Heisenbug). Fix: construct with `size_=0`
(and default-member-init as belt-and-suspenders).
